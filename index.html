Abu Dhabi University · Capstone Project · December 2025

NetSentinel

Automated Phishing Detection and DNS Security System

The challenge

Accessible DNS-layer protection

SMEs, students, and remote users often face phishing, DNS hijacking, and malicious redirection without the budget or specialist staff required for enterprise security platforms.

The solution

A unified endpoint security platform

NetSentinel routes DNS requests through a hardened resolver, evaluates domains against policies and threat intelligence, applies heuristic risk scoring, and displays live activity through a centralized dashboard.

< 2 secManual rule enforcement
28 msAverage DNS response
62 MBPeak endpoint-agent RAM
0.8%Idle endpoint-agent CPU

Key capabilities

  • Real-time DNS filtering through the Unbound resolver
  • Automated ingestion and merging of threat-intelligence feeds
  • AI-assisted detection using entropy, frequency, TTL, and reputation signals
  • Windows endpoint agent with DNS leak-prevention controls
  • Role-based dashboard, live analytics, audit logs, and policy management

My contribution

I co-developed the NetSentinel application and working prototype with Ameen Siddiqui. I also contributed with the full team to the project report and final presentation.

Secure RESTful Peer-to-Peer Skills Exchange Platform

Supporting peer-to-peer learning

Abu Dhabi University · CSE400 Network Security and Forensics

Evil Twin Attack

Raspberry Pi Rogue Access Point and Network Traffic Analysis

Authorised academic simulation This project was performed only in a controlled lab to study attack behaviour, evidence, and defensive controls.

The objective

Understand wireless impersonation risks

The experiment examined how a duplicate network can exploit user trust, redirect traffic, present a simulated captive portal, and expose sensitive communications when defensive controls are absent.

The simulation

A Raspberry Pi rogue access point

I configured a Raspberry Pi as the rogue access point, prepared the supporting DNS, DHCP, and HTTP services, connected a controlled test device, captured the generated traffic, and analysed the results.

1 RPiRogue access-point platform
4 layersDNS, DHCP, HTTP, and packet capture
2 toolstcpdump and Wireshark analysis
Lab onlyControlled and authorised environment

What the lab demonstrated

  • Rogue access-point behaviour and duplicate-network impersonation
  • DNS redirection toward a controlled captive portal
  • DHCP and HTTP service configuration in a lab environment
  • Packet capture of DNS queries and HTTP activity
  • Protocol and traffic analysis using Wireshark

My contribution

I completed the full practical simulation. I configured the Raspberry Pi as the rogue access point, built and hosted the simulated portal, prepared the network services, captured the experiment traffic, and analysed the outcome to understand how credentials and communications could be exposed.

Team

Mohamed Ibrahim Idris · Salih Samir

Defensive lessons

  • Verify public Wi-Fi names and disable automatic network connections
  • Use WPA2/WPA3 Enterprise with certificate validation where available
  • Deploy rogue-access-point monitoring and wireless intrusion prevention
  • Protect management frames and enforce HTTPS or VPN use
  • Harden endpoints against open or untrusted wireless networks

IoT Course Project · Raspberry Pi Security

Raspberry Pi Network IDS

Lightweight Network Monitoring and Threat Detection for IoT Devices

The challenge

Protecting resource-constrained IoT systems

Small edge devices can be exposed to scanning, SSH brute-force activity, ARP spoofing, and other network threats while operating with limited processing power and security tooling.

The solution

A lightweight Raspberry Pi NIDS

The system uses Python and Scapy to inspect live packets, identify suspicious TCP and ARP behaviour, write security events to logs, and provide firewall controls for managing blocked IP addresses.

5 threatsSSH brute force, SYN, XMAS, NULL, and ARP
LivePacket monitoring on a selected interface
ScapyPython packet inspection and detection logic
iptablesManual IP block and unblock controls

Detection capabilities

  • Potential SSH brute-force activity on port 22
  • Nmap SYN, XMAS, and NULL scan patterns
  • ARP poisoning through IP-to-MAC inconsistencies
  • Timestamped console alerts and persistent log records
  • Menu-driven monitoring and firewall-management controls

My contribution

I contributed to creating the IDS functionality and validating the system results. My work included helping develop the detection logic, testing the system with simulated network activity, reviewing generated alerts and logs, and confirming that supported attack patterns were identified correctly.

Collaboration

Repository collaborators: Ameen Siddiqui and Mohamed Idris.

Validation and accuracy

The IDS was evaluated with normal and simulated malicious traffic. Detection results were reviewed through terminal alerts and log entries. The current implementation contains iptables-based manual block and unblock controls; the scan-detection paths primarily alert and log rather than automatically blocking every detected source.

Technology stack

Raspberry Pi Python 3 Scapy Linux iptables Nmap SSH ARP TCP/IP IoT Security

Internet of Things 2 · TkCircuit Automation

Automatic Gate Control System

Distance-Based Servo Gate Automation with State-Machine Control

The objective

Automating safe gate movement

The project explores how an IoT controller can use distance information and explicit operating states to manage a gate without continuous manual input.

The solution

A virtual GPIO gate controller

A Python state machine controls a simulated servo motor and status LEDs through TkCircuit. Randomized values represent an ultrasonic distance sensor and trigger opening, waiting, and closing behaviour.

4 phasesClosed, opening, opened, and closing
< 2 mSimulated distance trigger for opening
0–180°Servo positions across the gate cycle
10 secConfigured open-state waiting interval

Key capabilities

  • Simulated ultrasonic-distance readings from zero to five metres
  • State-machine transitions for opening and closing sequences
  • Servo positions representing closed, moving, and fully open states
  • Separate LEDs for gate-open and gate-closed feedback
  • TkCircuit interface for testing GPIO behaviour without physical hardware

My implementation

I implemented the Python simulation, mapped the virtual GPIO components, developed the gate-state logic, and tested the transition sequence using different simulated distance values.

Design approach

The controller separates entry and stay behaviour for each gate phase so actuator commands and transition conditions remain clear and testable.

Internet of Things 2 · TkCircuit Monitoring

Temperature Monitoring & Emergency System

Temperature-Based Alerts, State Management, and Ventilation Control

The objective

Responding to changing temperatures

The project demonstrates how an IoT monitoring system can classify operating conditions and activate different outputs when temperature moves from normal to high or emergency levels.

The solution

A state-driven emergency controller

A TkCircuit simulation combines start and reset buttons, status LEDs, a buzzer, and a servo-operated ventilation mechanism. Generated temperature readings drive transitions between idle, monitoring, high-temperature, and emergency behaviour.

30°CHigh-temperature state threshold
> 40°CEmergency-state trigger
10 secAutomatic transition from idle to monitoring
180°Emergency ventilation-servo position

Key capabilities

  • Idle, monitoring, high-temperature, and emergency operation
  • Generated test temperatures across a 20–50°C range
  • Start and reset controls through simulated pushbuttons
  • System-on and emergency LED status indicators
  • Buzzer activation and ventilation-servo control during emergencies

My implementation

I created the virtual component configuration, implemented the Python state machine, defined the temperature thresholds, connected the alert outputs, and tested normal, high-temperature, emergency, and reset scenarios.

Simulation note

The course version uses randomized temperature values to validate system behaviour without requiring a physical analogue temperature sensor.
